This section explains the architectural patterns and protocols that power Vobiz telephony services. Understanding these concepts helps you make informed decisions about how to connect your application to the PSTN, which integration path fits your use case, and how different components interact at runtime.

- How SIP signaling and RTP media work — the full protocol mechanics, not just the surface
- How Voice XML WebSocket streaming connects phone calls directly to your AI pipeline
- When to choose SIP (LiveKit, VAPI, Retell, call transfer) vs. WebSockets (Pipecat, direct Python, cost optimization)
- How callbacks and webhooks enable real-time event notifications for call lifecycle events
- Configuration patterns for callback URLs, authentication, and hierarchy

## Choosing an architecture

The two primary integration paths for connecting phone calls to AI voice agents are **SIP trunking** and **WebSocket streaming**. They operate at completely different layers of the stack.

| | SIP trunking | WebSocket streaming |
|---|---|---|
| Protocol layer | Telephony (signaling + RTP media) | Application (TCP/WebSocket) |
| Setup complexity | High — trunk config, IP ACLs, SIP URI routing, firewall | Low/Medium — VoiceXML webhook + WebSocket JSON |
| Call setup latency | 1–5 seconds (INVITE handshake + PSTN routing) | Near-instant (WebSocket handshake + webhook fetch) |
| Infrastructure cost | Vobiz rate + AI platform fee (LiveKit/VAPI/Retell) | Vobiz rate + raw AI API costs only |
| Enterprise PBX | Native — Avaya, Cisco UCM, Teams Direct Routing | Not applicable |
| Best for | LiveKit, VAPI, Retell AI, ElevenLabs, call transfer | Pipecat, direct Python/Node.js, cost optimization at scale |

<Tip>
  Under 50,000 calls/month, a managed SIP platform (VAPI, Retell, LiveKit) saves weeks of engineering time. Above 50,000 calls/month, owning a WebSocket pipeline pays off by eliminating per-call platform markup.
</Tip>
